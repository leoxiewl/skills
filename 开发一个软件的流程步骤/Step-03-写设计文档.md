# Step 03 — 写设计文档

> [!abstract]
> 设计文档不是一次性写完的，是通过多轮讨论逐步长出来的。每次聊一个模块，AI 帮你落成文字，你来纠正和决策。

## 核心方法

**不要自己坐下来从头写。** 而是：

1. 告诉 AI 要写设计文档，说明项目背景
2. 先问"你想覆盖哪几块"——让 AI 列提纲，你来确认范围
3. 一块一块讨论，讨论完一块就写进文档
4. 每次 AI 写完，你来审查、纠正、补充细节
5. 反复迭代，直到你觉得可以动手写代码了

> [!tip]
> 设计文档的作用不是"记录你已经想清楚的东西"，而是**帮你把模糊的想法逼成具体的决策**。写的过程就是想清楚的过程。

## 设计文档应该包含哪些内容

根据项目类型按需取用，通常覆盖：

| 模块 | 内容 |
|---|---|
| 产品定义 | 核心问题、产品定位、MVP 功能边界、目标用户 |
| 技术选型 | 语言、框架、关键技术方案对比与决策 |
| UI 交互设计 | 状态流转、面板布局线框图、交互细节 |
| 数据结构 | 数据模型、存储方案、字段定义 |
| 待决策项 | 还没拍板的问题，留着后续讨论 |

> [!important]
> **MVP 边界要锁死。** 设计文档最重要的作用之一是把"以后再说"和"Day 1 必须有"分开，防止范围蔓延。

## 多轮讨论的节奏

```
第 1 轮：确定文档范围（问：要覆盖哪几块？）
第 2 轮：逐块讨论内容（每块聊清楚再写进文档）
第 N 轮：审查 AI 写的内容，纠正细节、补充决策
持续：有新想法 / 改了主意 → 直接改文档，保持文档是最新的
```

> [!warning]
> AI 会主动帮你扩展功能和细节，你的任务是**收窄**，不是跟着 AI 的思路走。每次讨论完问自己：这个是 MVP 必须有的吗？

## FocusBar 的设计文档迭代过程

本次经历的讨论轮次：

1. 确定文档覆盖范围（产品 / 技术 / UI / 数据结构，四块全要）
2. AI 生成初稿
3. **纠正**：MVP 三个区域明确（现在正在做 / 今日 Top 3 / 本周 Top 3）
4. **补充**：正在做的事情可以输入和编辑；Top 3 可以标记完成
5. **纠正**：正在做的事情不是唯一一条，支持多条并行
6. **调整**：面板改为左右两栏布局（左栏两个 Top 3，右栏正在做）

> [!note]
> 每次纠正都很小，但每次都让设计更精确。这种"AI 先出一版，我来纠正"的节奏比自己从头写快很多。

## 产出物

- `design.md`：放在项目目录根下（`focusbar/design.md`）
- 文档随讨论持续更新，不是写完就不动了
